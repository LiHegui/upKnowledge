⚛️ React 渲染行为 极详细图解

基于 Mark Erikson《A (Mostly) Complete Guide to React Rendering Behavior》

1 渲染两阶段流程：Render 阶段 → Commit 阶段
完整流程时序
① setState() / dispatch() — 触发渲染
React 将该组件标记为 "需要重新渲染"，放入调度队列
② Render 阶段 — 调用组件函数（纯计算，不碰 DOM）
从根部向下，递归调用所有被标记组件的函数/render()；
生成新的 React Element 树（即 JSX 对象）；
Reconciliation（diff）：新旧 Element 树对比，计算最小变更集
③ Commit 阶段 — 将变更写入真实 DOM（同步执行）
依据 diff 结果一次性操作 DOM；更新所有 refs
④ useLayoutEffect / componentDidMount / componentDidUpdate
DOM 已更新、浏览器绘制前同步执行（阻塞绘制）
⑤ 浏览器绘制（Paint）
用户看到新画面
⑥ useEffect（Passive Effects）— 异步执行
绘制完成后异步运行；先执行上次的 cleanup，再执行本次 setup
Render 阶段 vs Commit 阶段 详细对比

🔵 Render 阶段

  • 调用组件函数，生成新 Element 树
  • 内部执行 Reconciliation（diff）
  • 不修改任何真实 DOM
  • 可被中断、暂停、丢弃（并发模式）
  • 可多次执行同一组件
  • 渲染函数必须是纯函数

🟢 Commit 阶段

  • 拿到 Render 阶段计算好的变更列表
  • 一次性写入真实 DOM
  • 始终同步执行，不可中断
  • 只执行一次（已有确定结果）
  • 更新 refs
  • 触发 useLayoutEffect / 生命周期
⚠️ 常见误区：diff（Reconciliation）不是在两阶段"之间"，而是 Render 阶段内部的一部分！
生命周期执行顺序（首次挂载 vs 更新）
时机首次挂载后续更新
Render 阶段 调用组件函数 → 生成 Element 调用组件函数 → diff 新旧 Element
Commit — DOM 插入 DOM 节点 更新 DOM 节点
Commit — refs 绑定 refs 更新 refs
useLayoutEffect setup 运行 上次 cleanup → 本次 setup
浏览器绘制 Paint
useEffect setup 运行 上次 cleanup → 本次 setup
2 默认渲染行为：父组件渲染 → 子组件全部递归渲染
场景：B 组件 setState() 触发，组件树 A → B → C → D
  A SKIP — 未被标记
└─ B RENDER — setState() 触发
└─ C RENDER — 父级 B 渲染了
└─ D RENDER — 父级 C 渲染了
🔑 React 不关心 props 是否改变——只要父组件渲染，子组件就被无条件调用！
渲染 ≠ 更新 DOM：组件函数跑了，但 diff 后发现输出没变，DOM 不会改变。

6 组件类型（type）、Key 与协调
type 改变 → 销毁重建，type 不变 → 复用更新

❌ type 改变（销毁 + 重建）

旧：<div>
↓ type 从 div 变成 span
新：<span>
→ 销毁整棵 div 子树
从头重建 span 子树

✅ type 相同（复用 + 更新）

旧：<MyComp a=1 />
↓ props 变化，type 不变
新：<MyComp a=2 />
→ 复用现有实例
仅更新 props，不重建

❌ 渲染中创建新组件类型

// 错误示范 function Parent() { // 每次渲染都是新引用！ function Child() { ... } return <Child />; }
Key 的作用：身份标识 + 强制重建

❌ key=index（下标）— 问题多

初始：[A(0), B(1), C(2)]
删除 A，头部插入 X
[X(0), B(1), C(2)]
React 以为 key=0 的还是原来那个
错误复用 → 状态错位 🐛

✅ key=id（唯一ID）— 正确

初始：[A(id:1), B(id:2), C(id:3)]
删除 A，头部插入 X(id:4)
[X(id:4), B(id:2), C(id:3)]
React 精确识别：销毁 id:1，新建 id:4
✅ 状态正确，效率最优

💡 key 强制重置实例状态

// 选中项变化时彻底重置表单 <DetailForm key={selectedItem.id} /> // key 变 → 旧实例销毁 // → 新实例创建 → state 清空
7 渲染批处理：React 17 vs React 18
同一事件处理器中多次 setState
const onClick = async () => { setCounter(0); setCounter(1); // ← 与上一行合并 const data = await fetchSomeData(); // ← await 分割 setCounter(2); setCounter(3); // ← React 17: 两次独立渲染，React 18: 合并 };

React 17 — 3 次渲染

setCounter(0)
setCounter(1) ← 合并
→ Render #1
await 之后，批处理失效
setCounter(2)
→ Render #2
setCounter(3)
→ Render #3

React 18 — 2 次渲染（自动批处理）

setCounter(0)
setCounter(1) ← 合并
→ Render #1
await 之后，React 18 仍自动批处理
setCounter(2)
setCounter(3) ← 合并
→ Render #2

React 18 flushSync — 强制立即渲染

import { flushSync } from 'react-dom'; flushSync(() => { setCounter(0); // 立即渲染 }); // DOM 已更新 flushSync(() => { setCounter(1); // 再次立即渲染 });
8 状态快照：每次渲染的 state 是独立常量
为什么 setCounter 后立即读取还是旧值？
❌ 误解：JS 闭包？
// 普通 JS 闭包可以看到最新值 let counter = 0 const fn = () => { counter++ console.log(counter) // ✅ 1 }
≠
✅ 真相：每次渲染是独立 const
// React 每次渲染重新执行函数 function MyComp() { const counter = 0 // 本次渲染的常量！ const handleClick = () => { setCounter(counter + 1) // 告诉 React 下次用 1 console.log(counter) // 仍是 0，不会变 } }
🔑 setCounter 不修改当前的 counter，它只是告诉 React "下次渲染时用新值重新执行组件函数"。
两次渲染的 counter 是完全独立的两个 const，互不影响。
渲染快照模型（每次渲染 = 一帧独立的宇宙）
渲染 #1（counter = 0）
const counter = 0
return <div>{counter}</div> → "0"
handleClick 捕获的 counter = 0
→
点击
渲染 #2（counter = 1）
const counter = 1
return <div>{counter}</div> → "1"
handleClick 捕获的 counter = 1
→
点击
渲染 #3（counter = 2）
const counter = 2
return <div>{counter}</div> → "2"
handleClick 捕获的 counter = 2
9 渲染优化三剑客：React.memo + useMemo + useCallback
没有 memo vs 有 memo 的渲染流程

❌ 无 memo

👆父组件渲染
↓
🏃调用子组件函数（JS 开销）
↓
🔍diff 新旧输出
↓
😐输出没变 → DOM 不动（函数白跑了）

✅ 有 memo（props 未变）

👆父组件渲染
↓
🔎shallowEqual(newProps, prevProps)
↓
✅props 没变 → 直接跳过，函数不调用
三者关系：缺一不可
🛡️ React.memo 让子组件"可以"被跳过
+
🔗 useCallback 稳定函数引用
+
📦 useMemo 稳定对象/计算引用
=
🚀 有效跳过渲染 节省 JS 执行时间
⚠️ 缺少任意一环优化就失效！memo 有了但 props 每次是新引用 → memo 永远无法跳过。
完整示例：三者配合
const MemoizedChild = React.memo(ChildComponent); // 1. 包裹子组件 function ParentComponent() { // 2. 稳定函数引用（依赖为空 = 挂载后永不重建） const onClick = useCallback(() => { console.log('clicked'); }, []); // 3. 稳定对象引用 const data = useMemo(() => ({ a: 1, b: 2 }), []); return <MemoizedChild onClick={onClick} data={data} />; } // 效果：父组件重渲染时，onClick 和 data 引用不变 // → React.memo 的 shallowEqual 通过 → 子组件跳过渲染 ✅
引用相等性：为什么每次都是"新"引用？
// JavaScript 引用比较（=== 比的是内存地址） (() => {}) === (() => {}) // false — 两个不同的函数对象 ({ a: 1 }) === ({ a: 1 }) // false — 两个不同的对象 42 === 42 // true — 原始值比较值 "hi" === "hi" // true — 原始值比较值 // 每次函数组件执行，都会重新创建函数和对象 // 所以不用 useMemo/useCallback，memo 的 props 比较永远是 false
还有一种优化：相同元素引用（父组件控制）

方式一：props.children 透传

function Provider({ children }) { const [count, setCount] = useState(0); return ( <div> <button>{count}</button> {children} // 父组件渲染时 </div> // children 引用不变 ); // → 子组件不重渲染 }

方式二：useMemo 缓存 JSX

const child = useMemo(() => ( <ExpensiveChild /> ), [counter1]); // 只有 counter1 // 变化才重新渲染 return ( <div> <button onClick={() => setCounter2(c+1)}>C2</button> {child} // counter2 变时不重渲染 </div> );
10 不可变性与重渲染

❌ 直接 mutation（错误）

const onClick = () => { todos[3].completed = true; setTodos(todos); // 同一引用！ // useState 发现引用没变 // → 不触发重渲染 🐛 };

✅ 创建新引用（正确）

const onClick = () => { const newTodos = todos.slice(); newTodos[3].completed = true; setTodos(newTodos); // 新引用 ✅ // useState 发现引用变了 // → 触发重渲染 ✅ };
⚠️ useState / useReducer 需要新引用才会触发更新。this.setState() 不在乎，但 memo 等优化仍会因 mutation 而出现 bug。
11 Context 与渲染行为
Context 更新流程
① ParentComponent setState → 重渲染
生成新的 contextValue 对象（每次渲染都是新引用！）
② React 检查 MyContext.Provider 的 value prop
新引用 ≠ 旧引用 → 标记所有消费该 context 的组件需要更新
③ 所有 useContext(MyContext) 的组件强制重渲染
即使它们只关心 context 值的一部分，也会全部重渲染
④ 被重渲染的消费者，其子组件也递归渲染（默认行为）
React.memo + Context Provider 组合优化
ParentComponent setA(42) 触发
└─ MyContext.Provider value=新引用 Context 更新
└─ MemoizedChildComponent MEMO — no props changed → SKIP
└─ GrandchildComponent useContext → 强制渲染
└─ GreatGrandchildComponent 父级渲染 → 跟着渲染
💡 Sophie Alpert："Context Provider 正下方的组件应该用 React.memo 包裹"，这样父级状态更新不会影响不依赖 context 的子树。
Context 对象传入 Provider 的陷阱
// ❌ 每次 ParentComponent 渲染，contextValue 都是新对象 function ParentComponent() { const [a, setA] = useState(0); const contextValue = { a, b: 'text' }; // 每次新对象！ return <MyContext.Provider value={contextValue}>...</MyContext.Provider>; } // ✅ 用 useMemo 稳定引用，只有 a 变时才创建新对象 function ParentComponent() { const [a, setA] = useState(0); const contextValue = useMemo(() => ({ a, b: 'text' }), [a]); return <MyContext.Provider value={contextValue}>...</MyContext.Provider>; }

✓ 核心结论速查表
问题结论
父组件渲染，子组件会渲染吗？ ✅ 默认会，无论 props 是否变化
渲染 = 更新 DOM？ ❌ 渲染 = 调用组件函数；更新 DOM = Commit 阶段（可能不发生）
diff 在哪个阶段？ Render 阶段内部（Reconciliation），不是两阶段之间
setState 后立即读取 state 是新值吗？ ❌ 不是，每次渲染的 state 是独立常量，setX 只触发下次渲染
React.memo 能保证不渲染？ 需要配合 useMemo / useCallback 稳定 props 引用才有效
Context 更新会影响所有消费者吗？ ✅ 全部强制重渲染，目前无法按 selector 订阅
connect vs useSelector 哪个性能更好？ connect 自带防火墙效果；useSelector 需手动加 React.memo
到处加 memo 好吗？ ❌ memo 有比较开销，只在组件频繁以相同 props 渲染且开销大时才加
