Vue3 Diff 算法（快速 Diff） ​

参考：Vue3 源码解析 - diff 算法

一、什么时候触发 Diff？ ​

当组件重新渲染时，会生成新的 VNode 树。Vue 需要对比新旧两棵树的 children，找出最小的 DOM 操作来完成更新。

📊 Diff 触发流程
响应式数据变化→组件 render()→新 VNode 树→patch(旧VNode, 新VNode)→patchKeyedChildren
💡 patchKeyedChildren 就是核心 diff 算法所在。它只处理同一层级的子节点列表，不跨层级对比。

二、key 的作用 — 判断「是不是同一个节点」 ​

Diff 的第一步是判断两个 VNode 是否可以复用。判断依据是 type + key 都相同：

ts
function isSameVNodeType(n1: VNode, n2: VNode): boolean {
  return n1.type === n2.type && n1.key === n2.key
}
📊 有 key vs 无 key 的 Diff 差异
❌ 无 key / 用 index 做 key
旧列表
A
i=0
B
i=1
C
i=2
新列表（删除 A）
B
i=0
C
i=1
i=0: A≠B → 更新
i=1: B≠C → 更新
i=2: → unmount
结果：3 个节点全部操作
✅ 用唯一 id 做 key
旧列表

key=a 消失 → unmount(A)
B、C 通过 key 找到 → 复用
结果：仅 1 次删除，状态保留

三、核心五步 — 图解全流程 ​

Diff 的核心思路 ​

当响应式数据变化后，组件重新执行 render，产生一棵新的虚拟 DOM 树。现在有两棵树：旧的和新的，需要找出它们的差异，用最少的真实 DOM 操作完成更新——这就是 diff 要做的事。

但是两棵完整的树逐个对比太慢了（O(n³)），所以 Vue 做了一个约定：只在同一层级对比子节点列表，不跨层级。这样问题就简化为：给你两个数组（旧 children 和新 children），怎么用最少的操作把旧数组变成新数组？

Vue3 的策略是分五步，先处理简单的，最后处理复杂的：

📊 五步策略总览
1
从头扫描，跳过相同前缀
i 从左往右走，新旧相同就 patch 复用，不同就停下
2
从尾扫描，跳过相同后缀
e1、e2 从右往左走，新旧相同就 patch 复用，不同就停下
3
旧的扫完了，新的有剩余 → 挂载新增
条件：i > e1 && i <= e2
4
新的扫完了，旧的有剩余 → 卸载多余
条件：i > e2 && i <= e1
5
都没扫完 → 中间乱序，用 LIS 算法找最少移动方案
建 key→index 映射 → 遍历旧节点 patch/unmount → LIS 决定谁动谁不动
💡 大部分实际场景在前两步就处理掉了大半节点。只有复杂乱序才需要 Step 5。

📊 5c — LIS + 倒序遍历
LIS 计算：

倒序遍历（后面的先处理好，可以用作锚点）：

k === LIS[j=1] = 3 → 跳过，j--
k=2

k === LIS[j=0] = 1 → 跳过，j--
k=0

insertBefore(E, C的DOM) ← C 已在正确位置，用作锚点
DOM 变化过程：
① unmount(F)→... [C] [D] [E] ...
② insert(H) before D→... [C] [H] [D] [E] ...
③ move(E) before C→... [E] [C] [H] [D] ...
✅ C 和 D 全程没动，只操作了 3 次（1 删 + 1 插 + 1 移）。如果没有 LIS，需要移动 4 个节点。

为什么用 LIS？ ​

❌ 不用 LIS
逐个检查位置，不合适就移动
可能移动 n 次
很多移动是多余的
✅ 用 LIS
找到最长已有序序列 → 不动
只移动不在序列中的节点
LIS 越长 → 操作越少 → 越快

倒序遍历的原因 ​

DOM 的 insertBefore(node, anchor) 需要一个锚点。倒序保证后面的节点已就位，可以用作锚点：
k=3 D 不动（锚点是 G，已确定 ✅）
k=2 H 新增，insertBefore(H, D) ← D 已在正确位置 ✅
k=1 C 不动
k=0 E 移动，insertBefore(E, C) ← C 已在正确位置 ✅
⚠️ 如果正序遍历，后面的节点还没处理好，找不到正确的锚点！

moved 标记优化 ​

遍历旧节点时，如果新位置一直递增 → 说明顺序没变 → moved = false → 跳过整个 LIS 计算！
if (newIndex >= maxNewIndexSoFar) maxNewIndexSoFar = newIndex // 递增，OK
else moved = true // 位置变小了，有移动
💡 大部分更新场景顺序不变，这个优化命中率很高，直接省掉 O(n log n) 的 LIS 计算。

patched 计数优化 ​

维护 patched 计数器，新节点全部匹配完后，剩余旧节点不用查 Map，直接 unmount：

✅ p=1

❌

✅ p=2 满了!

直接删

Step 5 完整简化源码 ​

ts
// ======= Step 5: 乱序处理 =======
const s1 = i  // 旧中间起点
const s2 = i  // 新中间起点

// --- 阶段一：建 keyToNewIndexMap ---
const keyToNewIndexMap = new Map()
for (let j = s2; j <= e2; j++) {
  keyToNewIndexMap.set(newChildren[j].key, j)
}

// --- 阶段二：遍历旧节点 patch/unmount ---
const toBePatched = e2 - s2 + 1
let patched = 0
let moved = false
let maxNewIndexSoFar = 0

// newIndexToOldIndexMap: 新位置 → 旧位置+1（0 表示新增）
const newIndexToOldIndexMap = new Array(toBePatched).fill(0)

for (let j = s1; j <= e1; j++) {
  const oldVNode = oldChildren[j]

  if (patched >= toBePatched) {
    unmount(oldVNode)  // 提前退出优化
    continue
  }

  const newIndex = keyToNewIndexMap.get(oldVNode.key)

  if (newIndex === undefined) {
    unmount(oldVNode)  // 新列表没有这个节点，删除
  } else {
    newIndexToOldIndexMap[newIndex - s2] = j + 1  // 记录旧位置

    if (newIndex >= maxNewIndexSoFar) {
      maxNewIndexSoFar = newIndex
    } else {
      moved = true  // 顺序变了
    }

    patch(oldVNode, newChildren[newIndex], container)
    patched++
  }
}

// --- 阶段三：移动和挂载 ---
const increasingNewIndexSequence = moved
  ? getSequence(newIndexToOldIndexMap)  // 求 LIS
  : []  // 不需要移动，跳过 LIS 计算

let j = increasingNewIndexSequence.length - 1

// 倒序遍历新中间部分
for (let k = toBePatched - 1; k >= 0; k--) {
  const nextIndex = s2 + k
  const nextChild = newChildren[nextIndex]
  const anchor = nextIndex + 1 < newChildren.length
    ? newChildren[nextIndex + 1].el  // 下一个节点的 DOM 作为锚点
    : parentAnchor

  if (newIndexToOldIndexMap[k] === 0) {
    // 新增节点
    patch(null, nextChild, container, anchor)
  } else if (moved) {
    if (j < 0 || k !== increasingNewIndexSequence[j]) {
      // 不在 LIS 中 → 移动
      move(nextChild, container, anchor)
    } else {
      j--  // 在 LIS 中 → 不动，跳过
    }
  }
}

Vue2 双端 Diff 详解 ​

📊 Vue2 — 4 指针双端比较
Vue2 用 4 个指针，从新旧两个列表的头和尾同时向中间靠拢：
旧：
A

B

B

对比顺序比较命中后操作
① 头头os vs nspatch, os++, ns++
② 尾尾oe vs nepatch, oe--, ne--
③ 头尾os vs nepatch + 移动 os 到 oe 后面
④ 尾头oe vs nspatch + 移动 oe 到 os 前面
逐轮图解（D→B→A→C 的变换）：
第 1 轮：④ 尾头命中 D===D
patch(旧D, 新D) → 移动 D 到 A 前面 → oe--, ns++
DOM: D A B C
第 2 轮：② 尾尾命中 C===C
patch(旧C, 新C) → 位置不变 → oe--, ne--
DOM: D A B C（C 没动，只 patch）
第 3 轮：③ 头尾命中 A===A
patch(旧A, 新A) → 移动 A 到 oe 后面 → os++, ne--
DOM: D B A C
第 4 轮：① 头头命中 B===B
patch(旧B, 新B) → os > oe，结束！
DOM: D B A C ✅
⚠️ Vue2 双端 Diff 的局限：
▸ 没有预处理，每次都要做 4 次比较
▸ 4 次都不命中时退化为遍历查找，最坏 O(n²)
▸ 没有 LIS，不能保证最少移动次数
▸ 没有 PatchFlag，无法跳过静态节点

六、LIS 最长递增子序列 — 算法推演 ​

📊 什么是 LIS？
在一个数组中找到最长的递增子序列（不要求连续）：
输入: [3, 1, 4, 1, 5, 9, 2, 6]
LIS: [1, 4, 5, 9] 或 [1, 4, 5, 6]，长度 = 4
在 diff 中的含义：LIS 中的节点位置已经递增，不需要移动，只移动不在 LIS 中的。
📊 Vue3 getSequence 算法推演：getSequence([5, 3, 0, 4])
算法策略：贪心 + 二分查找 + 回溯，时间复杂度 O(n log n)
i=0
arrI = 5，result 最后一个值 arr[0]=5
5 < 5? NO → 不追加，二分查找也不替换
result = [0]
i=1
arrI = 3，arr[result末尾]=5
5 < 3? NO → 不追加
二分查找：3 < 5 → 替换！result[0] = 1（3 比 5 更适合做开头）
result = [1]
i=2
arrI = 0 → 跳过（0 表示新增节点，不参与 LIS）
i=3
arrI = 4，arr[result末尾]=arr[1]=3
3 < 4? YES → 追加！result.push(3)，记录前驱 p[3]=1
result = [1, 3] → 对应值 3 和 4（递增 ✅）
最终结果（回溯后）：
result = [1, 3] → 索引 1(C) 和 3(D) 不需要移动
💡 为什么需要 p 数组回溯？贪心替换过程中 result 的值可能不是最终 LIS（替换只是为了让 result 保持"可延伸"）。通过 p 记录前驱关系，从后往前回溯才能还原真正的 LIS。
📊 三种算法复杂度对比
算法复杂度说明
暴力枚举O(2ⁿ)检查所有子序列
动态规划O(n²)dp[i] = 以 i 结尾的 LIS 长度
贪心+二分O(n log n)Vue3 使用，追加 O(1) + 替换 O(log n)

八、面试回答模板（30 秒版） ​

🎤 面试模板
Vue3 的快速 diff 分五步：① 从头扫描跳过相同前缀 ② 从尾扫描跳过相同后缀，这两步能处理「头尾不变只是中间变了」的常见场景；然后看指针状态：③ 新多旧少就挂载 ④ 旧多新少就卸载；如果中间还剩乱序部分，⑤ 建 key→index 映射找到对应关系，求 LIS 最长递增子序列确定哪些节点不用动，只移动剩余的，保证最少 DOM 操作。配合编译期 Block Tree + PatchFlag，静态节点直接跳过不参与 diff。
