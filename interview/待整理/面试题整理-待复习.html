Q1：Node.js 服务处理 500MB 大文件 + 错误处理 ​

一个 Node.js 服务需要处理一个 500MB 的日志文件，API 调用失败时需要捕获错误并返回 500 响应，以下哪种方案能确保错误被正确处理且不阻塞事件循环？

查看答案

核心考点：Stream 流式读取 + 错误处理

❌ 错误做法：fs.readFile 一次性读取 → 500MB 全部加载进内存，事件循环被阻塞，内存溢出。

✅ 正确做法：使用 fs.createReadStream 流式处理：

js
const fs = require('fs')
const readline = require('readline')

async function processLogFile(filePath, res) {
  const stream = fs.createReadStream(filePath, { encoding: 'utf8' })
  const rl = readline.createInterface({ input: stream })

  try {
    for await (const line of rl) {
      await processLine(line) // 逐行处理，不占用大量内存
    }
    res.status(200).json({ success: true })
  } catch (err) {
    rl.close()
    stream.destroy()
    res.status(500).json({ error: err.message })
  }
}

关键点：

  • createReadStream 每次只读一小块（默认 64KB），不阻塞事件循环
  • 用 for await...of 逐行消费，异常时可正确捕获并关闭流
  • 务必在 catch 里销毁流（stream.destroy()），否则文件句柄不会释放

Q2：高频 mousemove 事件导致仪表盘卡顿 ​

在一个大型数据仪表盘应用中，高频触发 mousemove 事件导致 UI 渲染卡顿，事件处理函数需要访问响应式状态并执行复杂计算，以下哪种方案能在保证状态同步的同时最小化性能开销，避免内存泄漏？

查看答案

核心考点：事件节流 + requestAnimationFrame + 内存泄漏防范

✅ 最优方案：requestAnimationFrame 节流（与屏幕刷新率同步）

js
let rafId = null
let latestEvent = null

element.addEventListener('mousemove', (e) => {
  latestEvent = e  // 只记录最新坐标，不立即处理
  if (rafId) return
  rafId = requestAnimationFrame(() => {
    handleMouseMove(latestEvent)  // 每帧最多执行一次
    rafId = null
  })
})

// 组件销毁时清理，避免内存泄漏
onUnmounted(() => {
  if (rafId) cancelAnimationFrame(rafId)
  element.removeEventListener('mousemove', handler)
})

其他补充优化：

  • 空间索引（kd-tree / 网格索引）：将命中检测从 O(N) 降到 O(log N)
  • Canvas 像素检测法：为每个数据点用唯一颜色渲染到离屏 Canvas，鼠标移动时读像素 → O(1) 命中
  • 数据量超大时做抽样，Tooltip 提示"附近有 N 个点，放大查看"

错误选项特征：

  • 直接在 mousemove 里 setState / 修改响应式数据 → 每秒触发上百次重渲染
  • 用 setTimeout(fn, 16) 模拟节流 → 不精确，可能累积队列
  • 忘记 removeEventListener → 组件销毁后监听器残留，内存泄漏

Q3：IE11 + 现代浏览器可视化图表兼容性综合题 ​

开发一个同时支持 IE11 和现代浏览器的可视化图表项目，遇到以下现象：

  1. CSS Grid 在 IE11 下图表容器严重错位
  2. 大量数据下鼠标移动检测导致系统卡顿
  3. 部分动画在某些浏览器存在帧率下降
  4. 项目要求支持 Retina 屏和普通屏

如何解决？

查看答案

一、CSS Grid 兼容 IE11

IE11 的 Grid 是旧版草案实现，不支持 grid-template-areas、自动放置、gap（需写 grid-gap）。

css
/* ✅ 方案1：@supports 降级 Fallback */
.chart-container {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  gap: 16px;
}

@supports not (display: grid) {
  /* IE11 走这里 */
  .chart-container {
    display: flex;
    flex-wrap: wrap;
    margin: -8px;
  }
  .chart-item {
    flex: 1 0 calc(25% - 16px);
    margin: 8px;
  }
}

✅ 方案2：postcss-preset-env 设置 browsers: ['ie >= 11'] 自动降级
✅ 方案3：直接放弃 Grid，统一写 Flexbox（最稳妥）

二、鼠标移动检测卡顿

见 Q2，核心：requestAnimationFrame 节流 + 空间索引加速命中检测。

三、动画帧率低

常见原因：

  • 用了 top/left 触发重排，应改用 transform
  • IE11 不支持 will-change，动画合成层优化失效
  • 使用了 CSS transition 但属性触发了重排
css
/* ✅ 动画只用 transform 和 opacity */
.animated {
  transform: translateX(0);
  transition: transform 0.3s ease;
}
/* ❌ 避免 */
.animated {
  left: 0;
  transition: left 0.3s; /* 触发重排 */
}

IE11 降级：检测不支持时回退到 JS 动画（requestAnimationFrame）。

四、Retina 屏适配（Canvas 高清）

js
function setupCanvas(canvas) {
  const dpr = window.devicePixelRatio || 1
  const rect = canvas.getBoundingClientRect()
  canvas.width = rect.width * dpr
  canvas.height = rect.height * dpr
  canvas.style.width = rect.width + 'px'
  canvas.style.height = rect.height + 'px'
  const ctx = canvas.getContext('2d')
  ctx.scale(dpr, dpr)
  return ctx
}

Q5：TypeScript — any 类型 API 响应安全转换 ​

一个函数接受 any 类型的 API 响应，想要安全地转换为 { id: number; name: string } 类型对象，以下哪项是推荐做法？

查看答案

❌ 不安全：直接 as 断言

ts
const user = response as User  // 只是告诉 TS 相信你，运行时不做任何检查

✅ 推荐：类型守卫（运行时验证）

ts
interface User {
  id: number
  name: string
}

function isUser(obj: any): obj is User {
  return obj && typeof obj.id === 'number' && typeof obj.name === 'string'
}

function processApiResponse(response: any): User | null {
  if (isUser(response)) {
    return response  // 类型安全，TS 已收窄
  }
  return null  // 或 throw new Error('数据格式异常')
}

✅ 更严格：unknown 作为中间层

ts
function safeParse(data: unknown): User {
  if (typeof data === 'object' && data !== null) {
    const obj = data as Record<string, unknown>
    if (typeof obj.id === 'number' && typeof obj.name === 'string') {
      return { id: obj.id, name: obj.name }
    }
  }
  throw new Error('Invalid user data')
}

常见错误选项：

  • <User>response（旧语法，等价于 as，同样不安全）
  • Object.assign({} as User, response)（仍无运行时验证）

核心原则：安全转换 = 运行时类型检查 + 类型收窄（is 谓词）

Q6：TypeScript — 实现类型安全的深度合并配置对象 ​

如何实现一个 mergeConfig 函数，在合并配置对象时对所有嵌套属性进行深度合并，并保持 TypeScript 类型安全？

查看答案

要求：

  • 支持任意层级嵌套对象
  • 数组默认覆盖（不合并）
  • 后面的配置覆盖前面同名属性
  • 返回值类型正确推断
ts
// 工具类型：深度合并两个类型
type DeepMerge<T, U> = {
  [K in keyof T | keyof U]: K extends keyof U
    ? K extends keyof T
      ? T[K] extends object
        ? U[K] extends object
          ? DeepMerge<T[K], U[K]>
          : U[K]
        : U[K]
      : U[K]
    : K extends keyof T
    ? T[K]
    : never
}

// 实现
function mergeConfig<T extends object>(...objs: T[]): T {
  if (objs.length === 0) return {} as T
  if (objs.length === 1) return objs[0]

  const result = { ...objs[0] } as any
  for (let i = 1; i < objs.length; i++) {
    const obj = objs[i]
    for (const key in obj) {
      const val = obj[key]
      const existing = result[key]
      // 两者都是普通对象（非数组、非null）才深度合并
      if (
        val && typeof val === 'object' && !Array.isArray(val) &&
        existing && typeof existing === 'object' && !Array.isArray(existing)
      ) {
        result[key] = mergeConfig(existing, val)
      } else {
        result[key] = val  // 数组、基本类型直接覆盖
      }
    }
  }
  return result
}
ts
// 使用示例
const merged = mergeConfig(
  { server: { port: 3000, host: 'localhost' }, features: ['a', 'b'] },
  { server: { port: 8080, https: true }, features: ['c'], newField: 'hello' }
)
// merged.server → { port: 8080, host: 'localhost', https: true }
// merged.features → ['c']  ← 数组直接覆盖
// merged.newField → 'hello'

Q7：动态仪表盘 5万行表格 + 实时折线图，布局抖动 + 掉帧 ​

一个大型数据仪表盘有：实时折线图 + 5万行数据表格（1000+列），要求主区域动画保持 60fps，左侧导航栏固定宽度，容器高度自适应内容，目前存在布局抖动和动画掉帧，如何优化？

查看答案

根本原因分析：

  • 5万行 DOM 节点 → Layout/Paint 开销极大，每次数据更新或滚动都触发重排
  • 1000+ 列 → 大量事件监听器，内存高
  • 实时折线图 SVG 更新 → 频繁 DOM 操作阻塞主线程
  • 容器高度自适应 → 牵连周边元素重排（布局抖动）

一、表格：虚拟滚动（必须做）

jsx
// React：react-window
import { FixedSizeList } from 'react-window'
<FixedSizeList height={600} itemCount={50000} itemSize={40} width="100%">
  {({ index, style }) => <div style={style}>{rows[index].name}</div>}
</FixedSizeList>

// Vue：vue-virtual-scroller
<RecycleScroller :items="rows" :item-size="40" v-slot="{ item }">
  <div>{{ item.name }}</div>
</RecycleScroller>

水平方向 1000+ 列同样做水平虚拟滚动；用 position: sticky 固定表头。

二、实时折线图：Canvas 替代 SVG

js
// 不重新创建实例，只调用 setOption 增量更新
let rafId
function onNewData(data) {
  if (rafId) return
  rafId = requestAnimationFrame(() => {
    chart.setOption({ series: [{ data }] })
    rafId = null
  })
}

三、防止布局抖动

js
// ❌ 读写交叉 → 强制同步布局
for (let i = 0; i < 100; i++) {
  el.style.height = el.offsetHeight + 1 + 'px'
}

// ✅ 先读后写（批量）
const h = el.offsetHeight
for (let i = 0; i < 100; i++) {
  el.style.height = h + i + 'px'
}
  • 用 ResizeObserver 监听容器尺寸，不在滚动事件里读 offsetHeight
  • 左侧导航用 position: fixed 或独立滚动层，不影响主区域

四、其他

  • will-change: transform 提示浏览器提前建立合成层
  • 表格列较多时，用事件委托（一个监听器）替代每列单独绑定
  • 实时数据节流：WebSocket 推送过快时，前端用滑动窗口取最新值

Q8：Node.js 高并发事务回滚 — EventEmitter vs Promise ​

在 Node.js 中有三个异步操作（验证支付、扣库存、发通知）需要保证原子性，比较以下两种方案：

  • 方案A：用 EventEmitter 监听各操作事件并触发全局回滚
  • 方案B：嵌套回调顺序执行，出错时逐步回滚

哪种更好？（场景：高并发、200ms 低延迟要求）

查看答案

结论：两种都不推荐，最优解是 Promise.allSettled + 补偿事务

方案A（EventEmitter）的缺点：

  • 代码分散，难以追踪执行顺序
  • 高并发下每个请求创建新监听器，内存泄漏风险
  • 事件监听本身无并发优势，反而增加复杂度
  • 不符合 200ms 低延迟要求

方案B（嵌套回调）的缺点：

  • 回调地狱，可读性差
  • 顺序执行：支付 → 扣库存 → 发通知，总耗时 = 三者之和（慢）
  • 回滚逻辑散落各处，容易遗漏

✅ 最优方案：Promise.allSettled + 补偿回滚

js
async function handleOrder(orderId) {
  // 并发执行三个操作，任一失败都能拿到结果
  const [payResult, stockResult, notifyResult] = await Promise.allSettled([
    verifyPayment(orderId),
    updateInventory(orderId),
    sendNotification(orderId)
  ])

  const failed = [payResult, stockResult, notifyResult]
    .map((r, i) => r.status === 'rejected' ? i : null)
    .filter(i => i !== null)

  if (failed.length > 0) {
    // 补偿事务：对已成功的操作进行回滚
    await compensate(failed, orderId)
    throw new Error('事务失败，已回滚')
  }
}

关键优势：

  • 并发执行（非顺序），总耗时 ≈ 最慢的那个操作
  • allSettled 不会因单个失败而短路，能收集所有结果
  • 回滚逻辑集中，可测试

生产级方案：引入消息队列（MQ） + 幂等补偿，支持跨服务分布式事务（Saga 模式）。

Q9：Webpack/Vite 集成按需加载插件，以下哪些配置正确？ ​

大型企业前端项目希望在 Webpack/Vite 中集成插件实现按需加载（如 Element Plus、Vant 等组件库），以下选项哪些能实现目标？

  • A. 在配置文件中使用函数定义插件，并在 options 属性中设置按需加载的相关路径
  • B. 直接用 build 覆盖默认输出位置，不调用插件具体配置
  • C. 在 vite.config.js 中将插件添加到 plugins 数组，但不进行任何其他设置
  • D. 在 vite.config.js 中将插件添加到 plugins 数组，并通过特定配置项启用按需加载功能
查看答案

答案：A 和 D

✅ A：正确 — 在配置文件中通过函数定义插件并传入 options，是标准做法：

js
// babel.config.js（Webpack 项目）
module.exports = {
  plugins: [
    ['import', {
      libraryName: 'element-plus',
      libraryDirectory: 'es',
      style: true  // 按需引入样式
    }]
  ]
}

✅ D：正确 — 在 vite.config.js 中添加插件并传入配置项：

js
// vite.config.js
import Components from 'unplugin-vue-components/vite'
import { ElementPlusResolver } from 'unplugin-vue-components/resolvers'

export default {
  plugins: [
    Components({
      resolvers: [ElementPlusResolver()]  // 通过配置项启用按需加载
    })
  ]
}

❌ B：错误 — 覆盖构建输出位置与按需加载无关。

❌ C：错误 — 只添加插件到 plugins 数组而不配置，插件不知道要处理哪个库，无法生效。

记忆要点： 按需加载 = 插件 + 配置（告诉插件是哪个库、从哪里取样式），缺一不可。

Q10：社交平台信息流 100 万条动态，移动端 60fps 虚拟滚动 ​

开发社交平台信息流页面，100 万条动态，要求 60 帧流畅滚动，移动端出现滚动卡顿、内存超过 2GB，以下哪个方案最优？

  • A. 使用 requestAnimationFrame 渲染，通过绝对定位动态计算元素位置，实现虚拟滚动，结合 Intersection Observer
  • B. 一次性渲染所有数据，用 transform: translate3d 优化滚动
  • C. 使用 Intersection Observer 懒加载图片，保留所有 DOM
  • D. 使用 overflow: scroll + 后端分页，滚动到底部加载下一页
查看答案

答案：A

为什么 A 正确：

虚拟滚动是唯一能解决 100 万条数据的方案：

  • 只渲染可视区域内 10~20 条 DOM，内存从 2GB 降到几十 MB
  • 绝对定位 + transform: translateY 动态计算每条的位置
  • 占位元素撑开总高度（totalHeight = itemHeight × totalCount）
  • requestAnimationFrame 节流滚动事件，确保 60fps
js
// 移动端虚拟滚动核心
const ITEM_HEIGHT = 120
const BUFFER = 5

function getVisibleRange(scrollTop, containerHeight) {
  const start = Math.max(0, Math.floor(scrollTop / ITEM_HEIGHT) - BUFFER)
  const end = Math.min(total, start + Math.ceil(containerHeight / ITEM_HEIGHT) + BUFFER * 2)
  return { start, end }
}

移动端专项注意：

  • 用原生 overflow: auto，不要用 iScroll 等 JS 模拟滚动（移动端原生动量滚动更流畅）
  • 用 transform 而非 top（避免重排）
  • 图片用 Intersection Observer 懒加载，减少内存

为什么其他选项不行：

选项问题
B：全量渲染 + translate3d100 万 DOM 仍存在，内存依然爆炸（>2GB）
C：懒加载图片但保留 DOM只解决图片，DOM 数量没变，内存和重排依然严重
D：传统分页上拉加载不能实现"60帧流畅滚动"，加载时会有白屏/跳动
