技术要点整理（待复习2） ​

来源：真实面试回顾，共 4 道题，涵盖树形数据处理、测试方案、分布式任务调度、高并发线程池等方向。

Q1：嵌套任务树形查询 + 大数据量返回前端如何处理？ ​

场景：一个任务已存在子任务，多个任务嵌套。如何查询这个任务的完整结构并组装数据返回给前端？若返回数据量过大，应如何处理？可以从交互层面说说想法。

查看答案

一、树形结构查询与组装 ​

方案 A：递归查询（适合层级不深）

js
async function getTaskTree(taskId) {
  const task = await Task.findById(taskId)
  const children = await Task.find({ parentId: taskId })
  task.children = await Promise.all(children.map(c => getTaskTree(c.id)))
  return task
}

⚠️ 注意：递归查询会产生大量 N+1 SQL，层级越深性能越差，不适合深层嵌套。

方案 B：一次查出全量 + 内存组装（推荐）

js
// 1. 一次查询所有相关节点
const allTasks = await Task.find({ rootId: rootTaskId })

// 2. 内存中构建树
function buildTree(list, parentId = null) {
  return list
    .filter(item => item.parentId === parentId)
    .map(item => ({ ...item, children: buildTree(list, item.id) }))
}

const tree = buildTree(allTasks, null)

方案 C：数据库闭包表 / 路径枚举（适合频繁查询）

  • 维护 ancestor / descendant 关系表，一条 SQL 取出所有后代节点

二、大数据量返回前端的处理方案 ​

方案说明适用场景
分页懒加载只返回顶层节点，点击展开再请求子节点层级深、节点多
虚拟滚动前端只渲染可视区域的节点平铺展示大量数据
数据裁剪默认只返回前 N 层，提供「展开全部」按钮多数场景只需浅层
接口压缩开启 gzip / brotli 压缩响应体数据量大但结构固定
流式返回使用 SSE / Stream 分批推送实时生成场景

交互层面建议：

  • 默认展示 2～3 层，超出部分折叠并显示节点数量提示
  • 搜索时走后端全文检索，不在前端遍历整棵树
  • 节点数 > 1000 时切换为「平铺列表 + 面包屑路径」模式，避免树渲染卡顿

Q2：登录功能发生需求变更，如何做全量测试？（接口 + 前端页面） ​

查看答案

一、接口测试维度 ​

正常流程

  • 正确账号密码 → 返回 token，状态码 200
  • token 结构校验（包含 userId、过期时间等字段）

异常流程

  • 错误密码 → 返回 401，错误信息符合预期
  • 账号不存在 → 返回 404 / 401（视安全策略，不应区分账号和密码错误）
  • 连续错误 N 次 → 触发锁定或验证码逻辑

边界条件

  • 空账号 / 空密码
  • 超长字符串（防止注入）
  • 特殊字符（<script>、'OR 1=1）
  • 并发登录同一账号（多端互踢 or 共存）

Token 相关

  • 过期 token 请求接口 → 返回 401
  • 伪造/篡改 token → 拒绝
  • 刷新 token 流程是否正常

二、前端页面测试维度 ​

UI 验证

  • 表单校验提示是否即时响应（blur/submit）
  • loading 状态、按钮防重复提交
  • 错误提示文案是否符合设计稿

流程测试

  • 登录成功后跳转目标页是否正确
  • 登录后刷新页面是否保持登录态
  • 退出登录后 token 是否清除、页面是否跳转到登录页

兼容性

  • 不同浏览器、移动端页面布局是否正常
  • 第三方登录（OAuth）回调流程

自动化方案

接口测试：Postman Collection + Newman / Jest + supertest
E2E 测试：Playwright / Cypress 录制登录流程
单元测试：表单校验逻辑、token 解析工具函数

Q3：分布式定时任务如何设计？ ​

查看答案

核心问题：防止同一任务在多个节点重复执行 ​

方案一：分布式锁（适合轻量场景）

js
// 基于 Redis SETNX 抢锁
async function runWithLock(taskName, fn) {
  const lockKey = `lock:task:${taskName}`
  const acquired = await redis.set(lockKey, '1', 'NX', 'EX', 60) // 60s 超时
  if (!acquired) return // 其他节点已在执行
  try {
    await fn()
  } finally {
    await redis.del(lockKey)
  }
}

⚠️ 注意：锁超时时间需大于任务执行时间，否则任务未完成锁已释放，另一个节点会重入。

方案二：任务调度中心（推荐生产使用）

框架语言特点
XXL-JOBJava调度中心统一分配，执行器集群只有一台运行
Elastic-JobJava基于 ZooKeeper，支持分片执行
BullMQNode.js基于 Redis，支持队列和延时任务
AgendaNode.jsMongoDB 持久化，轻量

方案三：数据库乐观锁（任务表标记）

sql
-- 只有 CAS 成功的节点才能执行
UPDATE task SET status='running', worker='node-1'
WHERE id=1 AND status='pending'

其他设计要点 ​

  • 任务幂等：同一任务即使被触发两次，结果一致
  • 失败重试：记录失败次数，超过阈值告警，不无限重试
  • 任务历史：每次执行记录开始时间、结束时间、结果、异常日志
  • 监控大盘：任务积压数、平均执行时长、成功率

Q4：高并发下线程池崩溃如何处理？ ​

查看答案

一、线程池崩溃的常见原因 ​

原因说明
任务堆积提交速度 > 消费速度，队列爆满触发拒绝策略
任务执行时间过长线程长时间被占用，新请求无线程可用
核心线程数配置过小并发峰值超出线程池处理能力
任务内存泄漏每个任务创建大对象未释放，OOM

二、处理方案 ​

1. 合理配置线程池参数

// CPU 密集型：线程数 ≈ CPU 核数
// IO 密集型：线程数 ≈ CPU 核数 × (1 + 等待时间/计算时间)

corePoolSize    = CPU 核数
maximumPoolSize = corePoolSize × 2
keepAliveTime   = 60s
queueCapacity   = 200（根据业务峰值评估）

2. 选择合适的拒绝策略

策略行为适用场景
AbortPolicy（默认）抛异常快速失败，业务层兜底
CallerRunsPolicy由调用方线程执行允许降速，不丢任务
DiscardPolicy直接丢弃日志等允许丢失的场景
自定义放入降级队列/告警生产推荐

3. 熔断降级（Node.js 示例）

js
// 超过阈值时快速返回降级响应，不继续提交任务
if (pendingTaskCount > MAX_QUEUE) {
  return res.status(503).json({ message: '服务繁忙，请稍后重试' })
}

4. 监控与告警

  • 监控指标：活跃线程数、队列深度、拒绝次数、任务平均耗时
  • 告警阈值：队列使用率 > 80% 触发告警，> 100% 触发熔断
  • 接入 Prometheus + Grafana 可视化大盘

5. 任务隔离

  • 核心业务与非核心业务使用独立线程池，防止相互影响
  • 耗时任务异步化，主流程快速响应
