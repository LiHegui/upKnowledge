实时协同系统 ​

基础概念篇 ​

Q: 实时协同编辑的核心技术挑战是什么？ ​

A:

实时协同编辑的核心挑战是并发冲突问题：当多个用户几乎同时修改同一位置时（如 A 在开头插入文字，B 几乎同时删除了前几个字），系统必须保证：

  1. 每个用户的操作都不丢失
  2. 所有人最终看到完全一致的结果（最终一致性）

业界主要有两种解决方案：OT（操作转换） 和 CRDT（无冲突复制数据类型）。

Q: OT（操作转换）和 CRDT（无冲突复制数据类型）的核心区别是什么？ ​

A:

两者代表了解决并发冲突的不同哲学：

特性OT（操作转换）CRDT（无冲突复制数据类型）
核心思想将并发操作进行"转换"，使其在新状态下仍能正确执行设计特殊数据结构，保证并发操作无需协调即可自动合并
架构依赖中心化，需要中央服务器接收、转换、广播操作去中心化，客户端本地解决冲突，天然支持 P2P
离线支持对网络延迟敏感，离线支持较弱高延迟容忍，天然支持离线编辑，网络恢复后自动同步
优势技术成熟稳定，是 Google Docs 等大型产品的基石扩展性强，性能好，适合大规模用户和高并发场景
劣势扩展性受限，转换逻辑复杂，难以处理大量并发用户数据结构复杂，实现难度高，可能有更大的存储/内存开销
典型代表Google Docs、ShareDBYjs、Automerge、Fugue

⚠️ 前沿动态：剑桥大学教授 Martin Kleppmann 提出的 Eg-walker 算法尝试融合两者优点——拥有 CRDT 的轻量级内存和快速加载，同时具备 OT 快速合并长分支的能力，代表未来发展方向。

技术选型篇 ​

Q: 主流富文本编辑器框架有哪些？如何选择？ ​

A:

现代主流编辑器均采用"数据驱动"架构（第三代编辑器）：将文档内容抽象为独立数据模型，再渲染到 DOM，从而实现高度可控性。

编辑器核心特点协同能力适用场景
ProseMirror功能强大、基于 Schema 设计，对文档结构有极致控制力内置 prosemirror-collab 支持 OT；可通过 y-prosemirror 集成 Yjs复杂应用，对性能和文档结构有极致要求，上层封装 Tiptap 更易上手
Slate插件化架构，所有功能均可作为插件，灵活性极高，与 React 深度绑定不内置协同，需自行实现 OT 或集成 slate-yjs需要深度定制编辑体验的复杂 React 项目
Quill轻量、API 简洁，通过 Delta 格式表示内容，开箱即用可通过 quill-delta 配合 ShareDB（OT）或 Yjs（CRDT）实现协同快速集成、功能标准的场景，MVP 验证首选

选型建议：

  • 追求最强协同 + 离线支持 → ProseMirror/Tiptap + Yjs
  • 快速验证、简单需求 → Quill + Yjs/ShareDB
  • React 深度定制 → Slate + Yjs

Q: 协同编辑中 Yjs 和 ShareDB 如何选择？ ​

A:

Yjs（CRDT 流派）：

  • 高性能 CRDT 库，将文本、数组、Map 等转换为可无冲突合并的 CRDT 结构
  • 网络层无关，可与 WebSocket、WebRTC 等任意协议集成
  • 天生支持离线编辑和去中心化架构
  • 生态极强：为 ProseMirror、Quill、CodeMirror 等几乎所有主流编辑器提供适配器

ShareDB（OT 流派）：

  • 成熟的 OT 实时数据库同步框架，运行在 Node.js 后端
  • 与 JSON 结构深度集成，与 MongoDB 配合默契，通常搭配 WebSocket
  • 基于中心化设计，对服务器依赖强，离线支持能力有限
  • 适合已有 Node.js + MongoDB 后端、希望使用经过大量验证的 OT 方案的项目
推荐组合：
├── 新项目 / 要求离线 / 去中心化  →  Yjs
└── 已有 Node.js 后端 / 偏好 OT   →  ShareDB

进阶实现篇 ​

Q: 协同系统进阶 ​

A:

1. 光标同步与用户感知 ​

除内容同步外，实时显示协作者的光标位置和文本选区同样关键。Yjs 提供了 awareness 协议（基于 y-protocols/awareness.js），专门用于高效处理这类非文档内容的实时状态同步。

2. 离线支持与版本管理 ​

  • 离线：CRDT 的天然优势。利用浏览器 IndexedDB 存储本地操作，网络恢复后自动与服务器同步，Yjs 生态对此有完善支持
  • 撤销/重做：协同环境下更复杂。Yjs 提供内置 UndoManager 管理本地操作历史；全局范围的撤销需要更复杂的策略设计

3. 网络连接与协议选择 ​

协议特点适用场景
WebSocket双向持久连接，是实时通信最常用方案中心化协同架构，服务器转发模式
WebRTC点对点传输，无需中央服务器去中心化 / 无服务器协同（Yjs + WebRTC）

4. 性能与安全优化 ​

  • 操作批处理：高频事件（连续打字）会产生大量微小操作，合并成批次发送可大幅降低网络和计算开销
  • 传输安全：生产环境必须使用 WSS（WebSocket Secure） 或 HTTPS，对通信内容加密
  • 权限校验：服务端需进行严格的身份验证和权限校验，防止未授权访问

⚠️ 注意：协同系统中安全往往被忽视——确保 WebSocket 握手时验证身份 token，并在每次操作时校验用户对该文档的读写权限。
